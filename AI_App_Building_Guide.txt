  - Email notification on confirmation

ADMIN PANEL FEATURES:
  - View all upcoming sessions (past, tomorrow, upcoming)
  - Edit and block schedule
  - View all payments and clients
  - AI-generated business summary
  - Cancel and reschedule appointments

3. STEP 1: BUILDING THE FRONTEND (DEMO MODE)
================================================================================

PHILOSOPHY:
  Build a fully functional demo/template FIRST before adding any backend.
  All logins, payments, and bookings should work with any credentials
  while in demo mode.

INITIAL PROMPT GUIDELINES:
  - Framework: React
  - Styling: Modern, professional, impressive
  - Include light and dark theme toggle
  - Blue gradient accents
  - Specify all features needed (session types, calendar, login, forms, etc.)
  - IMPORTANT: End prompt with "create the app in complete demo mode"
    so AI doesn't try to connect real APIs without credentials

KEY FEATURES TO INCLUDE IN FIRST PROMPT:
  - Session duration selector (e.g., 1-hour / 2-hour)
  - Calendar with available time slots
  - User form (name, email, requirements)
  - Payment flow (redirect to PayPal)
  - Login and registration screens
  - Booking confirmation/success screen

RUNNING LOCALLY:
  - Open terminal in project folder
  - Run: npm run dev
  - Open the localhost URL in any browser
  - App auto-updates on code changes (no need to refresh manually)

ADDING ADDITIONAL FEATURES (e.g., Admin Panel):
  Example prompt: "Add admin view. Users can sign in with user@admin.com
  and any password (demo mode). Admins can see booked sessions sorted by
  past/tomorrow/upcoming. Include cancel/reschedule buttons, payments view,
  and clients view filled with demo data."

IMPORTANT: Do NOT click "Accept All" until you have tested the changes.
  - Test all buttons, tabs, and forms first
  - Use "Reject All" if something broke
  - Only accept once everything works correctly

4. STEP 2: DEPLOYING & PUBLISHING
================================================================================

WHY DEPLOY EARLY:
  - Test redirects and payment flows in a live environment
  - Share with clients or testers
  - Recommended by experienced developers before writing any backend

DEPLOYMENT WORKFLOW:
  1. Push code to GitHub
  2. Connect GitHub repo to Vercel (vercel.com)
  3. Connect a custom domain (optional but recommended)

PUSHING TO GITHUB:
  - Open GitHub Desktop
  - Add existing repository (find your app folder)
  - Click "Publish Repository"
  - Add a description (optional)
  - Your code is now live on GitHub

DEPLOYING WITH VERCEL:
  - Go to vercel.com → Projects → Add New → Project
  - Import your GitHub repo
  - Click Deploy (that's it!)
  - Share the deployed URL immediately

CONNECTING A CUSTOM DOMAIN:
  - In Vercel project settings → Add Domain
  - Enter your domain name
  - Add the provided DNS records to your domain registrar:
      - CNAME record (for www)
      - A record (for root domain @)
  - Vercel auto-verifies records (usually within ~30 seconds)

CONTINUOUS DEPLOYMENT (AFTER INITIAL SETUP):
  - Make changes in Anti-Gravity
  - Test on localhost
  - Open GitHub Desktop → review changes (red = removed, green = added)
  - Write a commit summary (e.g., "Adding logo and favicon")
  - Commit → Push
  - Vercel automatically deploys the new version

5. STEP 3: ADDING FIREBASE BACKEND
================================================================================

WHY FIREBASE:
  - Pre-built authentication (email/password, Google, anonymous)
  - Firestore database (no need to define tables/columns)
  - User role management
  - Storage if needed

FIREBASE SETUP STEPS:

  A) CREATE PROJECT:
     - Go to console.firebase.google.com
     - Create new project, give it a name
     - Disable Gemini (optional), enable Google Analytics

  B) ADD WEB APP:
     - Below project name → Add App → Web icon
     - Name your app
     - Copy the SDK config code (save it — needed for Anti-Gravity prompt)

  C) SET UP AUTHENTICATION:
     - Build → Authentication → Get Started
     - Enable: Email/Password
     - Enable: Google Sign-In (select support email)
     - Enable: Anonymous Sign-In (for guests who book without registering)
     - Templates → Add custom domain (prevents emails going to spam)

  D) SET UP FIRESTORE DATABASE:
     - Build → Firestore Database → Create Database
     - Leave default settings, use Production Mode
     - Configure Security Rules (see below)

FIRESTORE SECURITY RULES SETUP:
  Use Gemini Pro to generate rules. Describe your app logic:

  Example collections and access rules:
    - schedule: Admin can edit; all users can read
    - bookings: Anyone can create; admin and booking owner can edit
    - users/{userId}: User can read their own data; admin can read all
    - users/{userId}/bookings: Private booking details per user
    - users/{userId}/payments: Private payment details per user

  Admin identification: Use a "role" field set to "admin" in user documents.

  To manually set a user as admin:
    - Go to Firestore → users collection → find user document
    - Add field: role = "admin"
    - Refresh the app — admin dashboard will load automatically

CONNECTING FIREBASE TO YOUR APP (via Anti-Gravity):
  Prompt example:
    "Add Firebase authentication with email/password and Google sign-in.
    Include forgot password. After registration, show email verification
    screen. For anonymous users booking without signing in, use anonymous
    auth. Paste your SDK code below: [paste SDK config here]"

  NOTE: Firebase API credentials are safe to include in the frontend.
  Do NOT paste Gemini API keys or PayPal secrets in prompts.

DATABASE LOGIC TO PROMPT:
  User logic:
    - On register or anonymous booking → save user data to Firestore
    - On booking → save slot + user ID to bookings collection
    - Save personal details to users/{userId}/bookings
    - On cancel/reschedule → update Firestore
    - On payment → save to users/{userId}/payments

  Admin logic:
    - On sign-in → check if user has role: "admin"
    - Admin sees: schedule editor, all bookings, all clients, all payments
    - schedule collection is editable by admin, readable by all

6. STEP 4: FIREBASE CLOUD FUNCTIONS
================================================================================

WHY USE CLOUD FUNCTIONS:
  - Keep API keys (Gemini, PayPal, Resend) hidden and secure
  - Anyone can access frontend code via browser dev tools
  - Functions run on Google Cloud — fast, serverless, secure

PREREQUISITES:
  - Upgrade Firebase project to Blaze (pay-as-you-go) plan
  - Set a budget alert (e.g., $30) as a safety net

SETUP STEPS:

  1. Install Firebase CLI and login:
     Run: npx firebase-tools login
     (Type "y" to install, "no" to optional prompts, sign in with Firebase account)

  2. Initialize functions in your project:
     Run: npx firebase-tools init functions
     - Select existing project
     - Choose JavaScript language
     - Type "no" to ESLint
     - Type "yes" to install npm dependencies

  3. A "functions" folder will appear in your project with index.js
     — this is where all your cloud functions live.

HIDING API KEYS IN FIREBASE:
  Run: npx firebase-tools functions:secrets:set VARIABLE_NAME
  Examples:
    - npx firebase-tools functions:secrets:set GEMINI_API_KEY
    - npx firebase-tools functions:secrets:set RESEND_API_KEY
    - npx firebase-tools functions:secrets:set PAYPAL_CLIENT_ID
    - npx firebase-tools functions:secrets:set PAYPAL_SECRET
  You'll be prompted to enter the value (input is masked for security).

DEPLOYING FUNCTIONS MANUALLY:
  Run: npx firebase-tools deploy --only functions
  (Run from the root of your app, not inside the functions folder)

  To navigate back from functions folder: cd ..

8. EMAIL NOTIFICATIONS (RESEND API)
================================================================================

USE CASE: Send emails for bookings, confirmations, reminders, cancellations

RESEND SETUP:
  1. Create account at resend.com
  2. Copy your API key from the dashboard (shown on first page)
  3. Add and verify your domain under the Domains tab
     - You get access to any email alias (e.g., notifications@yourdomain.com)
     - Add the provided DNS records to your domain registrar
     - Verification typically takes ~30 seconds

INSTALLATION (in functions folder):
  Run: cd functions
  Run: npm install resend

HIDE API KEY:
  Run: npx firebase-tools functions:secrets:set RESEND_API_KEY
  Paste your Resend API key when prompted.

PROMPT TO ANTI-GRAVITY:
  "Write a callable Firebase function using my Resend API key to send
  email notifications when a user books, cancels, or reschedules an
  appointment. Include clean HTML email templates for each notification
  type. Include: time, date, name, and email in the notification.
  Deploy the function when done."

RESULT:
  - Professional-looking HTML emails sent from your domain
  - Personalized with name, date, and time
  - Sent automatically on booking events

EXTENDING EMAIL NOTIFICATIONS:
  Once Resend is set up, you can add new email types easily by prompting
  Anti-Gravity to trigger the send email function for new events
  (e.g., new lead submission, quote requests, reminders).

9. PAYMENT INTEGRATION (PAYPAL)
================================================================================

IMPORTANT NOTE:
  PayPal integration is complex. Research thoroughly before prompting.
  Use Gemini Pro to understand the flow before writing code.

RESEARCH STEP:
  Ask Gemini: "I have a React app with Firebase auth/database and Firebase
  Functions. I'm setting up one-time PayPal payments. Research the latest
  PayPal documentation and explain the setup step by step."

PAYPAL SANDBOX SETUP:
  1. Go to PayPal Developer (developer.paypal.com)
  2. Under "PayPal Checkout" → Install on custom-built website
  3. Copy your Sandbox Client ID and Secret
  4. Create sandbox test accounts (Testing Tools → Sandbox Accounts)

WEBHOOK SETUP:
  1. Prompt Anti-Gravity to create a Firebase function as a PayPal
     webhook endpoint (just log events — no payment logic yet)
  2. Deploy functions to get the endpoint URL
  3. In PayPal Developer → Apps & Credentials → Default App → Add Webhook
  4. Paste the Firebase function URL
  5. Subscribe to events:
     - PAYMENT.CAPTURE.COMPLETED
     - PAYMENT.CAPTURE.DENIED
     - PAYMENT.CAPTURE.REFUNDED
     - CHECKOUT.ORDER.APPROVED

FRONTEND SETUP:
  Install PayPal library:
    Run: npm install @paypal/react-paypal-js

  Wrap App.jsx with PayPalScriptProvider using your sandbox Client ID.
  Replace payment button with the official PayPalButtons component.

HIDE CREDENTIALS:
  Run: npx firebase-tools functions:secrets:set PAYPAL_CLIENT_ID
  Run: npx firebase-tools functions:secrets:set PAYPAL_SECRET

PAYMENT FLOW:
  1. User clicks PayPal or card button
  2. Frontend calls backend to generate an order ID
  3. Backend authenticates with PayPal (OAuth), creates order, returns ID
  4. User completes payment on PayPal
  5. Frontend marks payment as "pending" (do not confirm from frontend)
  6. PayPal webhook fires → backend updates Firestore → booking confirmed
  7. Only send confirmation email AFTER webhook confirms payment

SWITCHING TO LIVE PAYMENTS:
  - Replace sandbox credentials with live PayPal credentials
  - Update Firebase secrets with live Client ID and Secret

10. TIPS & BEST PRACTICES
================================================================================

DEVELOPMENT WORKFLOW:
  - Always test locally before accepting changes in Anti-Gravity
  - Use "Accept All" only after thorough testing
  - Use "Reject All" if a change broke something
  - Commit to GitHub with descriptive messages after every significant change

SECURITY:
  - NEVER put Gemini, PayPal, or other secret API keys in the frontend
  - Firebase SDK credentials are safe for the frontend
  - Always use Firebase Cloud Functions for third-party API calls
  - Use Firebase Secrets to store all sensitive API keys

AI MODEL SELECTION:
  - Use Gemini 3.1 Pro for complex design tasks (limited quota)
  - Switch to Claude or GPT OSS models when Gemini quota runs out
  - Avoid Gemini 3 Flash for complex tasks

BUILDING IN STAGES:
  Stage 1: Build complete frontend in demo mode (no real APIs)
  Stage 2: Deploy and test in live environment
  Stage 3: Add Firebase authentication and database
  Stage 4: Write and deploy cloud functions for advanced features

FIRESTORE TIPS:
  - No need to define tables/columns in advance
  - Use security rules as your backend logic
  - Identify admins with a simple "role" field in user documents
  - Store private data in subcollections (users/{userId}/bookings)

ANTI-GRAVITY TIPS:
  - Use Planning Mode for major features
  - Use Fast Mode for straightforward tasks
  - Always specify "deploy after writing" when asking for cloud functions
  - If function wasn't auto-deployed, run manually from terminal

WHEN STUCK:
  - For Firebase issues, review Firestore security rules
  - For payment issues, check PayPal webhook event logs
  - For function issues, check Firebase Cloud Functions logs
  - Research external API documentation thoroughly before prompting
